--------------------------------------------------------------------------------
  THE IDEA IN ONE PARAGRAPH
--------------------------------------------------------------------------------

Kant is a fully serverless, peer-to-peer encrypted messaging platform built
for people and organisations that refuse to trust a third party with their
communications. There are no servers to breach, no databases to subpoena, and no
company that can hand over your messages — because none of it is stored anywhere
central. Personal use is free and open source forever. Enterprises pay a yearly
licence for a hardened version that adds admin policy controls, remote wipe, and
member management — all without giving the admin any ability to read content.
Every message travels over onion-routed P2P connections. Every device is locked
behind a mandatory password. The code is fully auditable by anyone.


--------------------------------------------------------------------------------
  THE PROBLEM
--------------------------------------------------------------------------------

Every major messaging and team communication platform — Slack, Teams, WhatsApp,
Telegram, even many "secure" alternatives — stores data on centralised servers.
This creates a single point of failure for privacy:

  - Servers can be hacked (Slack 2015, Microsoft 2023, Twilio 2022)
  - Servers can be subpoenaed by governments
  - Employees at the company can access metadata or content
  - Advertisers can influence what the platform does with data
  - A company can change its privacy policy at any time

Even Signal, the gold standard for encryption, runs centralised servers and
stores metadata about who contacts whom. Self-hosted alternatives like Mattermost
and Rocket.Chat still require someone to run a server that becomes a target.

The root issue is architectural: centralisation is the vulnerability.


--------------------------------------------------------------------------------
  THE SOLUTION
--------------------------------------------------------------------------------

Kant eliminates the server entirely. Key principles:

  1. TRULY SERVERLESS
     There is no Kant server. Peers connect directly to each other using
     WebRTC and libp2p. Messages are never stored anywhere except on the
     sender's and recipient's own devices.

  2. ONION ROUTING
     All traffic is routed through multiple encrypted hops (like Tor) so that
     no peer can tell who is communicating with whom, even at the network level.

  3. SELF-SOVEREIGN IDENTITY
     Users have no account with Kant. Their identity is an Ed25519 keypair
     generated on their device. Their public key is their identity. No email,
     no phone number, no username database.

  4. DOUBLE RATCHET ENCRYPTION
     Every conversation uses the Signal Double Ratchet algorithm — the same
     cryptography that powers Signal. Every message gets a new encryption key.
     Compromising one message does not compromise past or future messages
     (forward secrecy + break-in recovery).

  5. MANDATORY APP PASSWORD
     The app cannot be opened without a password. The password derives the
     local decryption key via Argon2id. A seized device is useless without it.

  6. BLIND ADMIN FOR ENTERPRISES
     The enterprise version adds an admin node — a desktop app the company
     runs on an always-on machine. The admin can manage membership (approve
     joins, kick users, trigger remote wipes) but is cryptographically
     incapable of reading any message content. This is verifiable in the
     open-source code.

--------------------------------------------------------------------------------
  TECHNICAL ARCHITECTURE
--------------------------------------------------------------------------------

TRANSPORT LAYER
  - libp2p: unified transport abstraction over WebRTC + WebSockets
  - WebRTC for direct browser-to-browser connections
  - WebSockets as fallback through lightweight relay (relay sees encrypted
    onion packets only — no content, no metadata)
  - Noise protocol for channel-level encryption on every connection
  - STUN/TURN for NAT traversal only (no content passes through these)

ONION ROUTING
  - Messages are wrapped in multiple encryption layers before sending
  - Each hop decrypts one layer and forwards — no single peer sees both
    sender and recipient
  - Inspired by Tor's onion routing model, implemented over libp2p

IDENTITY
  - Ed25519 keypair generated locally on first app launch
  - Private key stored in device keychain / secure enclave, encrypted
    with Argon2id-derived key from user's app password
  - Public key is the user's identity — shared via QR code or invite link
  - No central user registry

ENCRYPTION
  - Double Ratchet Algorithm (Signal protocol) for all conversations
  - X3DH (Extended Triple Diffie-Hellman) for initial key exchange
  - AES-256-GCM for message payload encryption
  - Every message encrypted with a unique key — perfect forward secrecy

PEER DISCOVERY
  - Kademlia DHT (via libp2p) for global peer routing
  - mDNS for local network discovery (same office / WiFi)
  - Bootstrap nodes: stateless, open-source, hold zero data — just help
    new peers enter the DHT mesh

STORAGE
  - All messages stored in IndexedDB on user's device only
  - Media and attachments via Helia (browser-native IPFS)
    content-addressed, distributed across willing peers
  - Group shared state (online presence, read receipts) via GunDB CRDTs

ENTERPRISE ADMIN NODE
  - Electron desktop app (Windows + macOS + Linux)
  - Holds the organisation's Ed25519 admin keypair
  - Signs and broadcasts: org roster, join approvals, revocations,
    remote wipe instructions, policy objects
  - Never connects to the onion mesh — operates on a separate
    policy channel only
  - Cached policies are TTL-signed JSON objects stored on each device
  - When admin is offline: cached policies apply until TTL expires

REMOTE WIPE
  - Admin signs a wipe instruction: { action:"wipe", target_pubkey, timestamp, sig }
  - Instruction propagates through the mesh
  - Target device checks for pending signed instructions on every app unlock
  - If valid wipe instruction found: purges all local IndexedDB data,
    keypair, and cached policies before rendering any UI
  - Cannot be bypassed because decryption (and thus any UI) requires
    the password check, which triggers policy verification

GROUP KEY ROTATION ON KICK
  - When a member is removed, admin broadcasts a revocation
  - All remaining peers verify the admin signature and update local roster
  - A new group encryption key is distributed to all remaining members
  - The removed user's public key is excluded — they cannot decrypt
    any message sent after rotation
  - Combined with remote wipe: complete cryptographic and data severance

TECHNOLOGY STACK
  - Language:     TypeScript throughout
  - Transport:    libp2p (@libp2p/webrtc, @libp2p/kad-dht, @libp2p/noise)
  - Crypto:       libsodium-wrappers (Ed25519, X3DH, Double Ratchet)
  - Storage:      idb (IndexedDB wrapper), Helia (IPFS in browser)
  - State:        GunDB (P2P CRDTs for shared state)
  - UI:           React + Vite
  - Mobile:       Capacitor (wrap web app for iOS + Android)
  - Desktop:      Electron (admin node + optional desktop client)
  - Packaging:    vite-plugin-pwa (installable PWA for web/mobile)

--------------------------------------------------------------------------------
  WHAT TO BUILD FIRST — THE CRITICAL PATH
--------------------------------------------------------------------------------

Do not try to build everything at once. The single most important thing you can
ship is a working, trustworthy 1:1 encrypted P2P chat. Everything else is built
on top of that proof.

The order that matters:

  1. Keypair + password encryption     (trust foundation)
  2. P2P connection via libp2p         (the hard technical problem)
  3. Double Ratchet encryption         (the security core)
  4. Basic chat UI                     (makes it real)
  5. Open source release               (builds community and credibility)
  6. Group chat                        (expands utility)
  7. Admin node                        (unlocks enterprise revenue)
  8. Mobile apps                       (unlocks mass market)
  9. Security audit                    (unlocks serious enterprise)

Do not skip to step 7 or 8 before steps 1–4 are solid. An enterprise customer
discovering a cryptographic flaw after paying you is far worse than a slow launch.

--------------------------------------------------------------------------------
  RISKS AND HOW TO HANDLE THEM
--------------------------------------------------------------------------------

RISK: Adoption cold start (P2P needs people online to work well)
HOW:  Nail the invite flow. One person pulls in their whole team via
      invite codes. Target communities that already care (legal, medical).

RISK: Enterprise sales cycles are slow (6–18 months)
HOW:  Build runway. Keep personal expenses low. The open source community
      generates goodwill and inbound interest while you wait.

RISK: Competitor copies the concept
HOW:  Move fast. Open source code is not your moat — trust, reputation,
      and the audit are. Be first to get audited and publish the results.

RISK: Cryptography flaw discovered
HOW:  Get the audit done before serious enterprise sales. Bug bounty programme.
      Respond fast and transparently if anything is found — this builds more
      trust than having no bugs would.

RISK: App stores reject or restrict the app
HOW:  PWA works without app store. Web app is always the fallback.
      Frame the store submission carefully — this is a business tool,
      not an anonymity tool.

RISK: Legal pressure (governments don't like things they can't intercept)
HOW:  Open source means you cannot be secretly compelled to add backdoors
      (the code is public — a backdoor would be visible). Operate in a
      jurisdiction with strong software freedom protections.
